## P1 — Fix before a paid pilot

### B1. CSV/Excel import silently drops valid inventory rows

**`services/fileParser.ts:163`** — `if (vals.length < headers.length) return null;`

Any data row with **fewer cells than the header** is silently discarded. This is common in real dealer exports: trailing optional columns omitted, a blank final field, ragged CSVs. The vehicle just vanishes — no count, no warning. On the primary "upload your inventory" path, silent data loss is the worst kind.

**Fix:** pad short rows instead of dropping them — `while (vals.length < headers.length) vals.push("");` — and/or surface a "skipped N malformed rows" toast so it's never silent.

### B2. `dealer_guard` falls open when auth or dealer is missing

**`backend/pb_hooks/dealer_guard.pb.js:25, 33-34`**

```js
if (!auth) return e.next(); // line 25 — no auth → proceeds
const authDealer = auth.get("dealer");
if (!authDealer) return e.next(); // line 34 — user w/ empty dealer → proceeds
```

A defense-in-depth write guard should **fail closed**. As written, an unauthenticated request (if any collection's create/update rule is ever loosened) or a non-superadmin user whose `dealer` field is empty/misconfigured proceeds **without the `dealer` field being forced** — so the client's payload `dealer` wins. That's a cross-tenant write vector hiding behind the assumption that API rules always catch it.

**Fix:** for a non-superadmin with no resolvable dealer, reject (`throw new BadRequestError(...)` / 403) rather than `e.next()`. Only superadmin should be exempt.

### B3. AI proxy has no rate limiting

**`api/_lib/ai/routes.ts:367`** — `void auth; // reserved for per-dealer rate limiting`

Every authenticated user can call `lender-extract` / `deal-analysis` / `lender-enrich` as fast as they like. These hit metered provider APIs (OpenAI/Anthropic/Gemini) on **your** keys. One compromised or careless account = an unbounded bill.

**Fix:** per-user (and per-dealer) token-bucket on the metered routes. Cheapest: a small in-memory limiter keyed on `auth.userId` (you already resolve `auth` at line 359); durable: a `rate_limits` PB collection or a KV counter.

### B4. AI proxy buffers unbounded request bodies + no upload caps

**`api/_lib/ai/routes.ts:64-82`** (`readRequestBody`) and **`:28-32`** (`FilePayloadSchema`)

`readRequestBody` concatenates all inbound chunks into memory with no ceiling, and `FilePayloadSchema` only requires `base64Data` to be non-empty — no max length, no `mimeType` allowlist. A large or malicious PDF (or a flood of them) is a memory-exhaustion DoS and a direct cost amplifier (big payloads → big provider token bills).

**Fix:** cap the buffered body (e.g. reject > ~15 MB), cap `base64Data` length in the schema, and constrain `mimeType` to `application/pdf`.

## P2 — Correctness & polish

### B5. Custom tax rate silently drops the out-of-state transit fee

**`services/calculator.ts:59-71`** — the `outOfStateTransitFee` is only added inside the **else** (default-logic) branch. If a user sets a `customTaxRate`, the `defaultState !== "MI"` transit-fee logic is skipped entirely, so an out-of-state deal with a custom rate loses the transit fee. Almost certainly unintended.

**Fix:** compute `extraFees` (out-of-state transit) independently of whether a custom rate is in play.

### B6. Blank APR silently becomes 0%

**`services/calculator.ts:88`** — `const interestRate = Number(dealData.interestRate) || 0;`

If the rate field is empty, the payment is computed at **0% APR** (`principal / term`), producing a plausible-looking but materially wrong, too-low payment. On a desking tool this is a "presented the customer the wrong number" risk.

**Fix:** treat blank rate as `"N/A"`/unset and refuse to show a payment, or surface a clear "enter a rate" state — don't default to free money.

### B7. No money rounding in the calculator

**`services/calculator.ts`** (whole file — confirmed: no `toFixed`/`Math.round`)

Tax, OTD, amount-to-finance, and payment are raw floats, and the payment is computed off the **unrounded** OTD. Expect penny drift and figures like `$432.9999`. Round currency to cents at the calculation boundary so the financed amount and the displayed amount agree.

### B8. Inconsistent error contract between the two finance functions

**`services/calculator.ts:28`** — `calculateLoanAmount` returns `0` for invalid input while `calculateMonthlyPayment` returns `"Error"` (`:8-10`). Callers that switch on `"Error"` will mis-handle the loan-amount path. Pick one contract.

### B9. 500 responses echo internal error messages to the client

**`api/_lib/ai/routes.ts:112, 392`** — `sendError` serializes `error.message` for any thrown error, including unexpected 500s. Minor info disclosure (internal paths/messages). Return a generic message for ≥500; keep detail in server logs/Sentry.

### B10. Delimiter detection misses tab-delimited files

**`services/fileParser.ts:60-65`** — only comma vs semicolon. A tab-separated export (common from some DMS) parses into a single column and every row fails validation. Add `\t` detection.

### B11. Synthetic VIN fallback can collide and pollutes identity

**`services/fileParser.ts:196-199`** — missing VINs get `VIN-<stock>-<rowIndex>`. VIN is used as the identity key (favorites, dedup). Across two imports the same stock+row can regenerate the same synthetic VIN, or a synthetic could collide with a real one. Prefer a guaranteed-unique token (e.g. `crypto.randomUUID()`-based) and treat synthetic VINs as non-mergeable.

### B12. `as any` write bypasses type checking on lender tiers

**`components/LenderProfileModal.tsx:120`** — `(tiers[index] as any)[name] = value;` writes an arbitrary `name` onto a tier with no type guard. A wrong/renamed field name writes silently and never type-errors. (Also `:303, :475` cast events to `any`, and `AiLenderManagerModal.tsx:271, 277`.) Narrow these to a typed key union.
